# 任务复盘博客与发布

当用户希望把一个**已经完成的任务**整理成一篇博客文章时，使用这个 skill。尤其适用于用户关心以下内容的场景：

- 原始问题到底是什么
- 从第一条指令到最后成功，中间真实发生了什么
- 实际执行了哪些步骤
- 中途遇到了哪些报错、失败或阻塞
- 为什么某个工具、命令或方案是必要的
- 有哪些相关知识点值得顺手讲清楚
- 最后还要把文章保存到指定博客仓库，并提交、推送

这个 skill 的定位是：**任务完成后的复盘写作与发布**，而不是脱离真实执行过程的泛泛写作。

## 何时使用

- “写篇博客记录一下”
- “把刚刚这个问题解决过程写成博客”
- “复盘一下从开始到成功中间经历了什么”
- “解释一下你是怎么解决这个问题的，并发到我的博客”
- “整理成一篇技术博客，提交并推送到我的博客仓库”
- 用户同时想要**解释过程**和**发布文章**

## 工作流

```text
1. 根据真实工具输出和仓库状态，重建执行时间线
2. 识别用户真正的目标、关键步骤、阻塞点、修复方式和最终结果
3. 收集博客仓库格式证据（_posts 命名、front matter、写作风格、git 状态）
4. 解释理解这次任务所需的相关概念（工具、协议、架构边界）
5. 写出一篇基于真实过程、而不是虚构流程的文章
6. 按目标博客仓库的真实格式把文章写入正确位置
7. 校验文件结构，并执行当前环境下最接近的可用验证
8. 如果用户明确要求发布，则继续提交并推送博客仓库
```

## 不可妥协的规则

### 1. 必须基于真实证据写作

**不要**编造一个看起来更顺滑、但实际上没有发生过的“干净故事”。

文章必须建立在以下真实依据之上：

- 会话中的真实工具输出
- 真实遇到的阻塞与异常
- 真实执行过的命令或操作
- 真实的仓库状态与认证状态

如果某一部分是推断出来的，而不是直接观察到的，必须明确收口，不要冒充事实。

### 2. 区分“事实”与“解释”

文章里应当明确区分三层内容：

- **这次任务里真实发生了什么**
- **这类工作流通常会怎么做**
- **相关技术概念在一般意义上是什么**

不要把一次具体案例里的路径，写成普遍唯一的标准答案。

### 3. 解释真正有价值的技术边界

博客不应该只是按时间顺序记流水账，而要解释清楚让这个任务变得不简单的“隐藏边界”，例如：

- 本地 git 与远程 GitHub 平台的边界
- CLI、API、MCP 三条路径的区别
- 认证问题与仓库状态问题的区别
- 构建验证与文件级验证的区别
- MCP host 与 MCP server 的职责边界

### 4. 优先遵循用户博客仓库的真实格式

在写文章之前，必须先检查目标博客仓库，并遵循它真实存在的约定：

- `_posts` 所在位置
- 文件扩展名（`.md` / `.markdown`）
- front matter 字段
- 标题 / 副标题 / tag 风格
- 最近文章的语气和结构

### 5. 只有在用户明确要求时才发布

如果用户只要求产出文章内容，那就写完为止。
如果用户明确要求提交 / 推送，则继续执行 git 校验、commit 和 push。

## 检查清单

```text
- [ ] 根据对话和工具输出重建原始任务
- [ ] 识别用户表面指令背后的真实目标
- [ ] 按顺序列出实际执行步骤
- [ ] 记录阻塞、失败尝试及其解决方式
- [ ] 提炼值得解释的关键知识点
- [ ] 检查目标博客仓库结构和文章格式
- [ ] 按博客原生风格写文章
- [ ] 以正确的文件名和 front matter 写入 _posts
- [ ] 读回文件，确认内容真实落盘
- [ ] 运行当前环境下最接近的验证（能 build 就 build，不能 build 就明确说明缺什么）
- [ ] 检查 git status、commit 风格、当前分支和 upstream
- [ ] 只有在用户明确要求时才 commit 和 push
```

## 推荐写作结构

对于这种“复盘型博客”，优先采用下面的叙述顺序：

### 1. 开头：先说清楚真正的问题是什么

用简单语言解释用户表面的请求。

例如：

- “表面上是让我 push 一个文件夹，实际上中间牵涉到了本地 Git、GitHub 认证和远程仓库创建。”
- “这不是单纯的一条命令，而是一条跨本地环境、平台认证和发布链路的完整流程。”

### 2. 时间线：真实发生了什么

从第一条指令开始，按顺序写到最终成功。

应包括：

- 最先检查了什么
- 缺了什么前置条件
- 安装了什么
- 哪一步失败了
- 后来改用了什么办法
- 最后是什么真正解开了阻塞

### 3. 技术解释：为什么这些步骤有必要

把时间线转化成“读者能理解的原理”。

常见解释角度：

- 为什么只有 `git` 还不够
- 为什么引入了 `gh`
- 为什么认证会卡住仓库创建 / 推送
- 为什么 MCP server 不能自动替代 shell 工作流
- 为什么构建验证失败可能只是因为本地运行时缺失，而不是文章本身有问题

### 4. 概念部分：把相关工具讲清楚

必要时解释这些概念：

- `git`
- `gh`
- GitHub MCP Server
- MCP host
- OpenCode
- PAT / SSH / CLI 认证

解释要尽量实用，并和这次案例重新连接起来。

### 5. 结尾：给出可复用的经验模型

结尾不要只停在“这次搞定了”，而要告诉读者下次遇到类似问题时该怎么想。

例如：

- “先确认本地仓库状态，再确认 GitHub 认证，再决定是走 `git` 路径还是 `gh` 路径。”
- “把工具分层看清楚，很多自动化失败就不再神秘。”

## 文章必须覆盖的内容

产出文章时，确保至少包含以下 7 个部分：

```text
1. 用户最初表面上提出的问题
2. 这个问题背后真正的隐藏问题
3. 从开始到结束的执行步骤
4. 中途遇到的阻塞/失败以及如何解决
5. 相关技术概念的通俗解释
6. 最终结果是什么，仓库/环境发生了什么变化
7. 面向未来类似任务的可复用结论
```

## 博客仓库验证规则

在提交前，尽可能按照环境实际能力完成验证。

推荐顺序：

1. 读回文章文件
2. 检查 front matter 和文件名
3. 检查博客仓库 git status
4. 如果本地运行时存在，执行站点构建
5. 如果运行时缺失，要明确说明缺了什么，以及已经完成了哪些验证

不要在本机没有 Jekyll/Hugo/Next 运行时的情况下，假装构建已经通过。

## Git / 发布规则

如果用户明确要求发布：

1. 检查 git status、diff、最近提交风格、当前分支和 upstream
2. 让提交信息匹配该仓库现有的 commit 风格
3. 只围绕这篇文章及其直接相关修改创建聚焦提交
4. 推送到当前跟踪的远程分支
5. 返回 commit hash 和最终分支状态

如果 push 被以下问题阻塞：

- 身份认证失败
- 缺少远程仓库
- 网络失败

那就：

- 明确说出阻塞点是什么
- 不要声称发布成功
- 让仓库保持在一个诚实、可继续处理的状态

## 示例

```text
用户：写篇博客记录一下刚刚这个问题，并推送到我的博客仓库。

你应该：
1. 根据真实证据重建刚刚完成的任务
2. 解释用户最初想要什么，以及真正卡住的问题是什么
3. 先检查博客仓库格式，再开始写
4. 写一篇基于真实执行路径的复盘文章
5. 用正确的 front matter 和文件名写入 _posts
6. 做文件级验证和当前环境允许的构建/运行时验证
7. 提交并推送博客仓库
8. 返回文章路径、commit hash，以及任何验证层面的限制
```

## 好的最终总结应该包含什么

使用这个 skill 后，一个好的最终答复应至少包含：

- 文章写到了哪里
- 是否已经 commit 和 push
- 对应的 commit hash
- 已经完成了哪些验证
- 还存在哪些环境限制（例如缺少 Ruby/Bundler 无法跑 Jekyll build）

这样用户拿到的不只是“一篇已经发布的文章”，还有一份对验证边界清晰、状态诚实的交付结果。
